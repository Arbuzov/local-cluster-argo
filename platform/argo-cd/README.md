# argo-cd

Argo CD itself, deployed via the official `argoproj/argo-helm` chart. The
Application is **self-managing** — once Argo CD is bootstrapped, applying this
manifest hands ownership of the install over to Argo CD, which then upgrades
itself on every change to `spec.source.targetRevision` or the inline values.

Current target: chart **`9.5.21`** (Argo CD **`v3.4.3`**), repo
`https://argoproj.github.io/argo-helm`.

`application.yaml` is kept **comment-free by convention** — every rationale
that would otherwise live in a YAML comment is documented here instead.

## Out-of-band secrets

`configs.secret.createSecret: false` — the chart does **not** render or manage
`argocd-secret`. It is a fully out-of-band Secret in the `argo-cd` namespace
that you must create/maintain by hand. Argo CD reads it by name at runtime;
nothing in this repo references the values. Keys it expects:

| Key | Purpose |
| --- | --- |
| `dex.google.clientID` | Google OIDC client ID, resolved into `dex.config` via the `$dex.google.clientID` placeholder |
| `dex.google.clientSecret` | Google OIDC client secret, resolved via `$dex.google.clientSecret` |
| `admin.password` / `admin.passwordMtime` | local-admin bcrypt hash + mtime (the local `admin` account is disabled in practice — see below — so these are inert) |
| `server.secretkey` | JWT signing key; auto-generated by `argocd-server` if absent |
| `accounts.homepage.tokens` | API token(s) for the `homepage` apiKey account |

Create / update the secret **before** the first sync, otherwise OIDC login
breaks and (because the local admin is disabled) the UI becomes unreachable:

```sh
kubectl -n argo-cd create secret generic argocd-secret \
  --from-literal=admin.password='<bcrypt-hash-of-your-admin-password>' \
  --from-literal=admin.passwordMtime="$(date +%FT%T%Z)" \
  --from-literal=dex.google.clientID='<google-oauth-client-id>' \
  --from-literal=dex.google.clientSecret='<google-oauth-client-secret>'
```

Generate the bcrypt hash with `argocd account bcrypt --password '<plaintext>'`.

To add only the OIDC keys to an existing secret (e.g. when migrating from an
older install that inlined them in `dex.config`):

```sh
kubectl -n argo-cd patch secret argocd-secret --type=merge -p \
  '{"data":{"dex.google.clientID":"<b64>","dex.google.clientSecret":"<b64>"}}'
```

Because the chart no longer owns this Secret, it must **not** carry Argo's
tracking metadata (`argocd.argoproj.io/tracking-id` annotation,
`app.kubernetes.io/instance` label) — otherwise Argo flags it for pruning. A
freshly hand-created secret has none; if you adopt one that was previously
chart-managed, strip those:

```sh
kubectl -n argo-cd annotate secret argocd-secret argocd.argoproj.io/tracking-id-
kubectl -n argo-cd label   secret argocd-secret app.kubernetes.io/instance-
```

### Redis auth

Redis credentials come from the out-of-band `argocd-redis` Secret (key
`auth`). `redisSecretInit.enabled: false` because Redis is co-located and
trusted; we don't run the chart's `argocd-redis-secret-init` Job. To switch to
a chart-managed Redis password, re-enable `redisSecretInit` and drop the
out-of-band secret.

## Node placement — nothing is persistent

Argo CD here keeps **no persistent state**: there are no PVCs in the namespace,
and the bundled Redis is a pure **ephemeral cache** (a cold restart just
rebuilds it). Nothing therefore needs to be pinned to a particular node, so no
component carries a `nodeSelector` — the scheduler places the controller,
Redis and the rest on any node (the chart's default `kubernetes.io/os: linux`
still applies). Previously `controller` and `redis` were pinned to
`kube-master`; that pin was removed because it bought nothing without
persistent local data.

## Server, ingress and HPA

- `server.extraArgs`: `--insecure --rootpath=/argo-cd --basehref=/argo-cd` —
  the UI/API are served under the `/argo-cd` sub-path behind the nginx ingress
  on `dev.whitediver.keenetic.link`; TLS is terminated at the ingress.
- `server.disable.auth: false`, `server.dex.server.strict.tls: false` — Dex is
  reached in-cluster over plain TLS at `argo-cd-argocd-dex-server:5556`.
- `server.autoscaling.enabled: false` and `repoServer.autoscaling.enabled:
  false` — single-node-ish home lab; the chart defaults HPA to `max=5`, which
  does not fit, so we run fixed single replicas.
- `controller.resources`: requests `512Mi`, limit `1Gi`. The default `384Mi`
  OOM-killed the v3.4.x application-controller while it reconciled ~20
  Applications (a sustained OOM-loop that stops the whole cluster's health from
  settling). The controller idles ~150–200Mi but spikes during a full
  re-reconcile, so it needs the headroom. It only fits on `kube-master` (the
  Pi workers are ~1GB).

## OIDC / Dex

`configs.cm.dex.config` configures a single Google OIDC connector and
references the client credentials as `$dex.google.clientID` /
`$dex.google.clientSecret`. Argo CD resolves `$`-prefixed names against
`argocd-secret` at runtime, so the real values never appear in this repo.

`configs.cm.admin.enabled: "false"` — the local `admin` account is disabled;
the only working login is Google OIDC for `info@whitediver.com` (mapped to
`role:admin` via RBAC). If the OIDC keys are missing from `argocd-secret`,
there is **no** fallback login.

## RBAC

`configs.rbac.policy.csv` defines:

- `role:dev-readwrite` — read all apps, sync apps in the `dev` project, read the
  `dev` project.
- `role:homepage` — read-only on all apps, bound to the `homepage` service
  account (used by the homepage dashboard widget).
- `info@whitediver.com` → `role:admin`; default role `role:readonly`.

## Health customizations (Lua)

Two custom health checks live in `configs.cm`:

- **`...health.apiextensions.k8s.io_CustomResourceDefinition`** — always reports
  `Healthy`. CRD health computation can time out on large schemas (e.g.
  `applicationsets.argoproj.io`), which would otherwise block syncs.
- **`...health.Service`** — this cluster has no LoadBalancer controller (no
  MetalLB etc.). LoadBalancer-typed Services bind to the host IP via
  `spec.externalIPs`, so `.status.loadBalancer.ingress` stays empty and the
  default check would peg them at `Progressing` forever. The script treats a
  non-empty `spec.externalIPs` as a valid ingress signal and reports `Healthy`.

## Resource exclusions

`configs.cm.resource.exclusions` drops high-churn / noisy resource kinds from
Argo's watch set to cut watched-event volume and UI clutter:

- **`apiextensions.k8s.io/CustomResourceDefinition`** — avoids the Lua
  health-check timeouts on large CRDs (and is why CRDs aren't Argo-managed, see
  below).
- **`Endpoints`, `EndpointSlice`** (core + `discovery.k8s.io`) — created by the
  control plane.
- **`Lease`** (`coordination.k8s.io`) — internal.
- **Authn/Authz review kinds** (`authentication.k8s.io`, `authorization.k8s.io`)
  — `*SubjectReview`, `TokenReview`, `*AccessReview`, etc.
- **`CertificateSigningRequest`** (`certificates.k8s.io`) and
  **`CertificateRequest`** (`cert-manager.io`) — intermediate.
- **Cilium internals** — `CiliumIdentity`, `CiliumEndpoint`,
  `CiliumEndpointSlice`.
- **Kyverno reporting/intermediate** — `*PolicyReport`, `*Report`,
  `UpdateRequest`, etc.

## CRDs are NOT managed by Argo

Because `CustomResourceDefinition` is in `resource.exclusions`, Argo CD will
**not** apply the chart's CRDs during a self-sync. They must be updated
out-of-band whenever the chart version bumps. Render them from the target chart
and server-side apply (server-side avoids the last-applied-config size limit on
the large ApplicationSet CRD):

```sh
helm template argo-cd argo/argo-cd --version <ver> -n argo-cd \
  -f <values> --include-crds \
  | yq 'select(.kind == "CustomResourceDefinition")' \
  | kubectl apply --server-side --force-conflicts -f -
```

CRD changes across recent versions are additive (e.g. v3.4.x adds an optional
`status.health` field to the ApplicationSet CRD), so they are
forward/backward compatible and need no rollback.

## Slow egress / repo-server exec timeout

The cluster's egress to GitHub Pages (`argoproj.github.io`,
`grafana.github.io`, …) goes through a slow VLESS tunnel. On a **cold cache**
(e.g. just after the repo-server pod restarts) `helm pull` of a chart can
exceed the repo-server's default `ARGOCD_EXEC_TIMEOUT=90s`, leaving apps stuck
in `ComparisonError: ... DeadlineExceeded`.

This timeout is deliberately **not** baked into this manifest. If it bites
after an upgrade (the repo-server rolls and loses its on-disk cache), live-patch
it; after one successful pull the chart is cached and the default 90s is enough
again:

```sh
kubectl -n argo-cd set env deployment/argo-cd-argocd-repo-server ARGOCD_EXEC_TIMEOUT=300s
```

## Upgrading the chart

1. Find the latest chart: `helm repo update argo && helm search repo argo/argo-cd --versions`.
2. Render-diff old vs new with the live values to confirm only the image tag
   (and additive CRD deltas) change; a clean render also validates the values
   against the new schema.
3. Apply the new CRDs out-of-band (see *CRDs are NOT managed by Argo*).
4. Bump `spec.source.targetRevision`, then `kubectl apply -f application.yaml`.
   Argo's automated sync rolls all components to the new app version. Watch the
   repo-server pull for egress timeouts (above).
